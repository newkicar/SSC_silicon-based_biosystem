# 报错救援（Error Rescue）

> 当错误反复出现、常规修复无效时，启动报错救援流程。
>
> 目标：跳出"改错→再错→再改"的死循环，从根因入手解决问题。

---

## 何时使用

| 场景 | 触发 |
|------|------|
| 同一个错误反复出现 | 用户说"一直修不好" |
| 改了又出现 | 修复后错误回归 |
| 不确定改了什么导致错误 | 错误来源不明 |
| 多个错误连锁爆发 | 修 A 出 B，修 B 出 C |

---

## 流程

### Phase 1: 停手（Stop）

1. **停止一切修改**：暂停所有代码改动
2. **记录当前状态**：
   - 最近的修改记录（git diff / git log）
   - 当前错误信息（完整 traceback / stack trace）
   - 最后一次正常工作的时间点

### Phase 2: 还原（Revert）

1. **逐层还原**：
   ```bash
   # 从最近一次正常状态开始
   git revert <last-good-commit>
   # 或手动恢复到已知正常的代码版本
   ```

2. **验证还原结果**：
   - 还原后错误是否消失？
   - 是 → 错误确实在最近引入
   - 否 → 错误不在最近修改中，进入 Phase 3

### Phase 3: 隔离（Isolate）

1. **最小复现**：
   - 剥离无关代码，找到触发错误的**最小代码片段**
   - 创建一个独立的测试文件验证

2. **二分法定位**：
   - 将代码一分为二，哪一半包含 bug？
   - 继续二分，直到定位到具体函数/行

### Phase 4: 根因分析（Root Cause）

1. **错误分类**：

| 错误类型 | 典型特征 | 分析方向 |
|---------|---------|---------|
| 语法错误 | SyntaxError, ParseError | 拼写、缩进、括号匹配 |
| 类型错误 | TypeError | 数据类型不匹配 |
| 值错误 | ValueError | 数据内容不合法 |
| 导入错误 | ImportError, ModuleNotFoundError | 依赖、路径、包名 |
| 运行时错误 | AttributeError, KeyError, IndexError | 对象状态、数据结构 |
| 逻辑错误 | 无报错但结果不对 | 算法、条件判断 |
| 环境错误 | 版本不兼容、路径问题 | Python/Node 版本、依赖 |

2. **深入分析**：
   - 阅读错误信息的**每一行**（包括 traceback）
   - 检查相关变量的**实际值**
   - 确认依赖版本的**兼容性**

### Phase 5: 修复（Fix）

1. **最小修复**：只修根因，不顺手改其他代码
2. **添加防护**：在修复点附近添加断言或日志
3. **添加测试**：为这个 bug 写回归测试

### Phase 6: 验证（Verify）

1. **正向验证**：原始错误是否消失
2. **回归验证**：相关功能是否正常
3. **执行 verify-changes.md**

## 输出格式

### 成功救援
```
✅ 报错救援完成。

**错误根因**：[一句话说明]
**修复内容**：[具体修改]
**验证结果**：[通过/未通过]
详情已记录到 memory/progress.md
```

### 无法自主解决
```
⚠️ 报错救援未能定位根因。

**已尝试**：
1. [步骤1]
2. [步骤2]
3. [步骤3]

**当前状态**：[描述]

**建议**：
- [建议1]
- [建议2]

请确认是否继续，或说「我自己来」。
```

## 与 bugfix-workflow 的关系

| 维度 | bugfix-workflow | error-rescue |
|------|-----------------|-------------|
| 适用场景 | 单一 bug，有明确修复方向 | 反复出错，死循环 |
| 复杂度 | 低 | 高 |
| 前置条件 | 项目能启动 | 项目可能完全不能启动 |
| 关系 | 先 baseline-startup → 再 bugfix | 如果 bugfix 循环 3 次以上 → 切换到 error-rescue |

---

## 自动切换条件

当以下信号出现时，建议从 bugfix 切换到 error-rescue：

1. 同一个文件连续修改 ≥ 3 次
2. 修一个 bug 引出 ≥ 2 个新 bug
3. 错误信息每次略有不同但核心相同
4. 用户明确表示"烦了"、"搞不定"等情绪信号
