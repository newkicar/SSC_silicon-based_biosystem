# 任务风险分级

> 借鉴 Sliver Vibe Coding 的风险分级思想，避免小任务过度治理、大任务跳过真源。

---

## 四级风险分级

| 风险等级 | 说明 | 治理成本 | 示例 |
|---------|------|---------|------|
| **普通工程任务** | 普通 bug 修复、局部 UI 调整、单文件改动、routine refactor、lint/type/test/build 修复、代码审查 | 最低：窄查、窄改、窄验 | 修一个 TypeScript 报错、按钮颜色调整、单测失败修复 |
| **轻量任务** | docs/copy/静态资源/样式-only 编辑，不改变产品行为 | 低：scoped read-edit-validate | 改 README、换海报、修错别字、调整局部样式 |
| **标准任务** | 功能或流程变更，需要真源证据、当前阶段计划和用户确认 | 中：完整治理 | 加一个新功能、改一个页面流程、改一个 API |
| **高风险任务** | 涉及架构、数据库、权限、支付、第三方、部署、安全、私有关键资料 | 高：强制治理 | 改数据库 schema、加支付功能、改权限系统、部署上线 |

---

## 升级规则

如果普通工程任务或轻量任务碰到以下情况，**自动升级**：

| 升级条件 | 升级后等级 |
|---------|-----------|
| 涉及非技术项目方向 | 标准任务 |
| 缺失/陈旧/冲突的真源文档 | 标准任务 |
| 所有权或架构不确定性 | 标准任务 |
| 数据库、权限、支付、第三方、部署、安全 | 高风险任务 |
| 重复 AI 修复循环 | 高风险任务 |
| 假成功、mock 数据、重复所有者 | 高风险任务 |

---

## 执行指引

### 普通工程任务
- 说明这是普通工程任务
- 检查相关文件
- 窄改
- 运行相关测试/检查
- 报告证据
- **不需要**：项目真源文档、阶段计划、广泛审计、用户确认

### 轻量任务
- 使用 scoped read-edit-validate loop
- **不需要**：当前阶段实施真源

### 标准任务
- 需要真源证据
- 需要当前阶段计划
- 需要用户确认需求

### 高风险任务
- 强制完整治理流程
- 需要架构判断
- 需要用户确认
- 需要实施真源
